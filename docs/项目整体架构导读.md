# 项目整体架构导读

> 适用范围：`agent-learning` 顶层架构说明、模块定位说明、后续多智能体与复杂工作流演进说明。

## 🎯 项目愿景与架构哲学

`agent-learning` 的目标，不是堆砌零散 Demo，而是系统性复刻《从零开始构建智能体》中的核心范式，并把“范式本体”与“企业级工程化落地”放进同一套仓库中长期对照。

项目的核心方法论是 **“左右互搏、对照学习”**：

- 左手：用纯手写版拆开底层状态流转与执行流，真正看清 Agent Runtime 的本质。
- 右手：用 Spring AI Alibaba 的高阶抽象，把同一范式重构为可扩展、可治理、可观测的企业级实现。
- 中轴：两条线都必须统一依赖项目既有的 `AgentLlmGateway` 防腐层，保证范式演进与模型接入解耦。

| 维度 | 探究本质（纯手写版） | 企业级落地（框架版） |
| --- | --- | --- |
| 核心目标 | 还原范式原貌，理解运行时到底怎么跑 | 验证范式如何稳定落地到生产工程 |
| 关注重点 | `while` 循环、状态机、消息接力、终止条件、显式路由 | `SupervisorAgent`、`Handoffs`、`FlowAgent`、`StateGraph`、A2A 等高阶能力 |
| 学习价值 | 看到“底牌”，避免只会调用框架 API | 看到“工程面”，避免停留在玩具 Demo |
| 约束边界 | 必须显式暴露状态、路由、终止逻辑 | 必须使用框架原生抽象，不能退化回普通手写流程 |
| 统一原则 | 两个版本解决同一道问题 | 两个版本都必须走统一 `AgentLlmGateway` |

一句话概括本项目：**用手写版理解范式本体，用框架版理解企业落地，用统一 Gateway 守住工程边界。**

## 🏗️ 核心基座说明

当前仓库是一个多模块 Maven 工程，技术基线已经明确为：`Java 21`、`Spring Boot 3.5.9`、`Spring AI 1.1.2`、`Spring AI Alibaba 1.1.2.0`。

当前根 `pom.xml` 已正式纳入 `module-multi-agent-roleplay`、`module-multi-agent-conversation`、`module-multi-agent-supervisor`、`module-graph-flow-paradigm`、`module-multi-agent-engineering` 五条多智能体与复杂工作流主线。

从顶层结构看，整个项目可以分为四层：

| 层次 | 模块 | 核心职责 | 稳定边界 |
| --- | --- | --- | --- |
| 核心协议层 | `framework-core` | 定义统一的消息、记忆、工具、LLM 请求与响应协议 | `AgentLlmGateway`、`LlmRequest`、`LlmResponse`、`Message`、`Memory`、`ToolRegistry` |
| 自动装配层 | `framework-llm-autoconfigure` | 基于统一 `llm.*` 配置自动选择 Provider 并装配网关 | 只向上暴露统一 `AgentLlmGateway` Bean |
| 模型适配层 | `framework-llm-springai` | 基于 Spring AI 完成具体 Provider 适配与能力映射 | 把底层模型差异收敛到统一协议之下 |
| 范式实验层 | `module-*` | 落地不同 Agent / Workflow / Multi-Agent 范式 | 只消费统一协议，不关心底层 Provider 细节 |

### `AgentLlmGateway` 为什么是铁律

`AgentLlmGateway` 是整个项目的统一模型门面，也是所有模块共同遵守的防腐层边界。

| 规则 | 含义 |
| --- | --- |
| 上层统一入口 | 所有范式模块只认 `AgentLlmGateway`，只面向 `LlmRequest -> LlmResponse` 编程 |
| 底层实现可替换 | 上层不需要感知底层接的是 Spring AI、OpenAI Compatible，还是未来的新 Provider |
| 能力统一收口 | 同步对话、流式输出、结构化输出都经由 Gateway 统一暴露 |
| 防止技术泄漏 | 禁止业务模块直接依赖 `ChatModel`、厂商 SDK 或私有接口 |
| 保证范式纯度 | 模块讨论的是 Runtime 和协作范式，而不是某家模型 SDK 的调用细节 |

因此，项目的工程边界非常明确：**范式模块研究“怎么组织智能体”，而不是“怎么直连模型”。**

## 🧩 多智能体与复杂工作流全景图

下表是项目当前多智能体协作与复杂工作流的全局映射。它回答的不是“目录怎么起名”这么简单，而是：**每个模块到底在复刻哪一种理论范式，以及它在 Spring AI Alibaba 中如何落地。**

| 模块 | 对应理论范式 | 纯手写版要揭示的运行时本质 | Spring AI Alibaba 落地手段 | 模块定位与价值 |
| --- | --- | --- | --- | --- |
| `module-multi-agent-roleplay` | `CAMEL` | 两个或多个互补角色围绕共享任务持续接力；显式维护角色契约、活动角色、共享 transcript、回合推进、终止协议与受控交棒 | 以 `Handoffs` 与控制权转移模式为核心，让活动 Agent 发生真实切换；必要时结合 `FlowAgent + StateGraph` 固化受控交接回环 | 负责沉淀“角色扮演 + 动态交棒 + 回环协作”主线，是理解 CAMEL 本质与企业 handoff 治理的关键模块 |
| `module-multi-agent-conversation` | `AutoGen` | 多专家围绕共享会话连续群聊；手写维护发言轮询、共享消息历史、回合上限与终止条件 | 以 `FlowAgent + StateGraph + shared_messages` 为核心，让多个节点围绕同一份群聊历史循环协作 | 负责沉淀“对话驱动协作”能力，适合开放式讨论、自动化群聊、多人分析与轮询推进 |
| `module-multi-agent-supervisor` | `Supervisor` | 中心化监督者负责全局拆解、动态分发、回收结果、继续决策或结束流程；手写版本质是一个 Supervisor Loop | 原生使用 `SupervisorAgent` 作为主执行抽象，由监督者统一调度 Worker，并在多轮循环中持续掌握全局控制权 | 负责沉淀“中心化治理”能力，适合企业级多步骤任务编排、分工协作与结果收敛 |
| `module-graph-flow-paradigm` | `LangGraph` | 用显式状态对象、节点、边、条件边、循环边构建运行时；手写版本质是图驱动状态机而非单纯 `if/else` | 基于 `FlowAgent + StateGraph API` 构建底层图编排机制，把复杂流程提升为可审计、可观测的状态图 | 负责沉淀“复杂工作流底座”，适合多分支、多循环、长生命周期、强可观测流程 |
| `module-multi-agent-engineering` | `AgentScope` | 关注消息驱动、状态隔离、上下文裁剪、分布式通信协议；手写版本质是显式消息网络而不是隐式上下文拼接 | 规划上以集成 A2A 智能体间通信协议为主要落地手段，支撑跨 Agent、跨进程、跨服务的工程化协同 | 负责沉淀“生产级多智能体工程能力”，面向强治理、高并发、长链路、分布式协作场景 |

> 说明 1：当前根 `pom.xml` 已正式纳入 `module-multi-agent-roleplay`，本文档与仓库现状保持一致。  
> 说明 2：`module-multi-agent-engineering` 当前已具备消息驱动与上下文工程语义，后续可继续沿 A2A 方向演进为更完整的分布式 Agent 通信模块。

### 配套导读入口

- `module-multi-agent-conversation`：详见《[AutoGen多智能体群聊与SpringAIAlibaba从0到1掌握指南](./AutoGen多智能体群聊与SpringAIAlibaba从0到1掌握指南.md)》，适合从 `FlowAgent`、`StateGraph`、`OverAllState` 和共享群聊状态流转角度理解 Spring AI Alibaba 版 AutoGen Conversation。

### 推荐学习路径

为了让学习路径与模块复杂度相匹配，建议按“先单体、再图流、后多智能体”的顺序推进：

| 阶段 | 推荐入口 | 学习目标 |
| --- | --- | --- |
| 第一阶段 | `module-react-paradigm` | 先建立最基本的 Agent Runtime 直觉，理解推理、行动、观察闭环 |
| 第二阶段 | `module-plan-replan-paradigm`、`module-reflection-paradigm` | 理解“先规划再执行”与“先生成再反思”的增强型单 Agent 范式 |
| 第三阶段 | `module-graph-flow-paradigm` | 理解为什么复杂流程最终要上升到显式状态机和图编排 |
| 第四阶段 | `module-multi-agent-conversation`、`module-multi-agent-supervisor`、`module-multi-agent-roleplay`、`module-multi-agent-engineering` | 系统掌握去中心化群聊、中心化调度、角色扮演交棒、分布式工程化协作 |

这个顺序的核心逻辑是：**先理解单个智能体如何形成闭环，再理解多个智能体如何共享状态、交接控制权、被统一调度，最后再进入工程化与分布式协同。**

## 🚀 开发与学习规约

为了让项目长期保持“可学、可比、可扩、可治理”，未来所有新模块与新范式都应遵守以下规约：

| 规约 | 强制要求 | 目的 |
| --- | --- | --- |
| 双版本对照 | 每个核心范式都必须同时提供纯手写版与框架版 | 避免只会调框架，不理解范式本体 |
| 同题对照 | 两个版本必须解决同一道业务题，输入输出目标保持一致 | 保证比较的是运行时设计，而不是题目难度 |
| Gateway 铁律 | 所有模块必须统一依赖并调用现有 `AgentLlmGateway`，禁止绕过网关直连模型 | 守住防腐边界，避免 Provider 细节污染上层范式 |
| 手写版义务 | 必须显式体现状态、路由、终止、消息传递与错误处理逻辑 | 让学习者真正看见 Runtime 的底牌 |
| 框架版义务 | 必须优先使用 Spring AI Alibaba 的原生抽象落地，不能把框架版退化成普通 `while` | 让企业级能力真正成为工程资产 |
| 状态显式化 | 所有多智能体与工作流模块都必须明确状态模型、上下文边界与交接协议 | 保证流程可审计、可调试、可扩展 |
| 文档先行 | 每个模块 README 都应清楚写明：理论范式、运行机制、手写版本质、框架版映射、Gateway 边界 | 保证项目结构一眼可懂，形成连续学习路径 |
| 演进一致性 | 新模块命名必须能映射到明确理论范式与明确框架落点 | 避免模块名、理论名、实现方式三者脱节 |

### 面向后续开发的落地要求

- 任何新范式进入仓库前，先回答清楚三个问题：它复刻的理论范式是什么，它的手写版要揭示什么本质，它在 Spring AI Alibaba 中的企业级映射是什么。
- 任何框架版实现，都应优先复用 `SupervisorAgent`、`Handoffs`、`FlowAgent`、`StateGraph`、共享状态、A2A 等高阶能力，而不是把框架只当成模型调用壳。
- 任何模块如果脱离 `AgentLlmGateway` 单独直连模型，都视为破坏项目架构边界。
- 任何文档如果只讲 API 用法、不讲状态流与控制流，也不符合本项目的学习哲学。

## 结语

`agent-learning` 的顶层设计，本质上是一套 **“统一模型边界之上的范式实验场”**：底层用 `AgentLlmGateway` 收敛模型接入，上层用多模块承载不同智能体范式，中间用“手写版 vs 框架版”的持续对照，把抽象理论转化为可验证、可复用、可工程化的知识体系。

这意味着，项目未来不是简单增加更多 Demo，而是沿着这条主线持续演进：**每新增一个范式，都要同时回答它的理论归属、运行时本质、企业级映射以及统一 Gateway 边界。**
