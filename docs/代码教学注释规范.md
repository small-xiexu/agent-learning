# 代码教学注释规范

> 适用范围：`agent-learning` 仓库内所有以“帮助 Java 工程师理解 Agent Runtime”为目标的源码、测试与示例代码。
> 目标读者：有 Java / Spring Boot 经验，但对 Agent 运行时、工作流编排、多智能体协作仍处于入门阶段的工程师。

---

## 1. 这份规范解决什么问题

这个仓库不是普通业务项目，它承担的是**范式学习 + 运行时拆解 + 工程映射**三重职责。

因此，代码注释不能只停留在“这段代码做了什么”，而必须进一步解释：

- 这段代码为什么存在
- 它在当前范式里扮演什么角色
- 它和手写版 / 框架版 / 相邻类之间是什么关系
- 它在运行时如何驱动状态流转、路由判断和终止条件

一句话概括：

**本仓库的注释目标，不是把代码翻译成中文，而是把运行时机制讲明白。**

---

## 2. 第一原则：优先解释机制，不要只解释结果

低价值注释通常只是在复述代码表面行为，例如：

```java
// 设置搜索关键词
state.setSearchQuery(searchQuery);
```

这种注释对初学者帮助很有限，因为真正的问题不是“这行设置了什么”，而是：

- 为什么要有 `searchQuery`
- 谁会读取它
- 它是业务结果还是中间桥接字段
- 不写回这个状态，后续哪一层会断掉

更符合本仓库风格的写法应该是：

```java
// 把“问题理解”阶段提炼出的关键词写回共享状态，
// 供下游 SearchNode 消费，作为“语义理解 -> 工具调用”的桥接字段。
state.setSearchQuery(searchQuery);
```

判断标准只有一句：

> 一个只懂 Java、不懂 Agent Runtime 的读者，能不能只靠注释理解这段代码为什么成立？

如果不能，注释就还不够教学化。

---

## 3. 第二原则：优先解释边界、切换点和协作关系

本仓库最容易让初学者看不懂的，不是某一行 API，而是这些地方：

- 手写版和框架版的对照关系
- 状态从哪里写入、被谁读取
- 路由决策为什么在这里发生
- 为什么这里用 `FlowAgent`，那里却是手写 `while`
- 为什么一个字段是业务数据，另一个字段只是运行时控制标记
- 为什么流程会在这里结束，而不是继续下一轮

所以注释优先级应该是：

1. 边界
2. 状态流转
3. 路由切换
4. 终止条件
5. 设计意图
6. 最后才是局部实现细节

---

## 4. 注释形式约定

### 4.1 声明级别：优先用 Javadoc

类、方法、字段这类“声明级别”的说明，优先使用：

```java
/**
 * ...
 */
```

适用对象：

- 类
- 接口
- 枚举
- 字段
- 方法

### 4.2 方法体内部：优先用普通块注释或行注释

方法体内部的执行逻辑说明，优先使用：

```java
// ...
```

或

```java
/*
 * ...
 */
```

不建议在方法体内部大量使用 Javadoc 风格注释：

```java
/**
 * 这里做状态切换
 */
if (...) {
}
```

原因很简单：

- `/** ... */` 天然更像“声明说明”
- 方法体内部更适合解释“运行逻辑”
- 教学型注释追求的是清晰，不是形式越正式越好

### 4.3 选择规则

- 说明“这个东西是什么”：用 `/** ... */`
- 解释“这段代码为什么这样跑”：用 `//` 或 `/* ... */`

## 5. 类级注释规范

本仓库的核心类，类注释至少要回答 5 个问题：

1. 它是什么
2. 它解决什么问题
3. 它在运行时扮演什么角色
4. 它的上游输入来自哪里
5. 它的下游输出会被谁消费

推荐模板：

```java
/**
 * xxx
 *
 * 职责：...
 * 运行时角色：...
 * 上游输入：...
 * 下游输出：...
 * 对照关系：...
 *
 * @author xiexu
 */
```

### 本仓库特别需要加强类注释的类型

- `Handwritten*Agent` / `*Coordinator`
- `*FlowAgent`
- `*Node` / `*NodeAction`
- `*Router`
- `*State` / `*StateKeys`
- `*GatewayBackedChatModel`
- `*PropertySupport`
- 用来承载“范式关键行为”的测试类

---

## 6. 方法级注释规范

本仓库的方法注释，不应只写“执行 xxx 逻辑”，而应该说明运行时语义。

推荐回答这几个问题：

- 谁会调用这个方法
- 它是入口还是中间步骤
- 输入是原始输入、共享状态，还是上游阶段产物
- 返回的是最终结果、阶段结果，还是写回状态后的中间事实
- 这个方法执行后，下一步通常由谁继续消费

推荐模板：

```java
/**
 * 执行 xxx。
 *
 * @param xxx 输入从哪里来、表示什么
 * @return 返回给谁消费、代表哪个阶段结果
 */
```

### 本仓库特别需要写清的方法

- `run(...)`
- `invoke(...)` 外层封装方法
- `process(...)` / `apply(...)`
- 路由判断方法
- 状态提取 / 状态转换方法
- Demo 测试中的 `createApplicationContext()`

---

## 7. 字段级注释规范

字段是初学者最容易“看得见名字、却看不懂用途”的地方。

对关键字段，不要只写“搜索结果”“最终答案”“当前状态”，而要明确：

- 谁写入它
- 谁读取它
- 它是业务事实，还是运行时控制字段
- 为什么必须把它显式放进状态里

### 推荐写法

例如 `GraphState.searchQuery`：

```java
/**
 * UnderstandNode 提取的搜索关键词，供 SearchNode 使用。
 */
private String searchQuery;
```

这个写法比“搜索关键词”更好，因为它额外交代了：

- 上游写入者
- 下游读取者
- 字段在流程中的桥接作用

### 本仓库必须重点解释的字段类型

- 共享状态字段
- 路由控制字段
- 终止标记
- 阶段输出字段
- `outputKey` 对应的状态键
- `iterationCount` / `maxTurns` / `recursionLimit` 一类护栏字段

## 8. 代码块注释规范

不是每个代码块都要注释，但下面这些块前面应优先加说明：

- 构造器链式调用
- 默认依赖注入
- 状态切换
- 条件边 / 路由分支
- handoff / 交接逻辑
- 循环、递归上限、终止条件
- 手写版和框架版的一一映射

### 例 1：构造器链式调用

本仓库里这类写法很多，例如：

```java
public HandwrittenGraphFlow(AgentLlmGateway gateway) {
    this(gateway, new MockSearchTool(false));
}
```

这种地方建议补成：

```java
// 提供默认构造方式：调用方只传 gateway 时，内部自动注入“正常成功”的搜索工具。
// 测试或演示失败分支时，可改用重载构造器手动注入 shouldFail=true 的搜索工具。
public HandwrittenGraphFlow(AgentLlmGateway gateway) {
    this(gateway, new MockSearchTool(false));
}
```

### 例 2：状态切换

这类地方不要只写“设置状态”，而要写：

- 为什么切状态
- 切完以后谁会读这个状态
- 这个状态会把流程带到哪里

### 例 3：条件路由

路由注释应明确解释：

- 决策依据是什么
- 返回值是什么语义
- 框架层会把这个返回值映射到哪个节点

## 9. 本仓库特有的解释重点

这是项目内规范和通用注释 Skill 最大的区别：**本仓库有自己特有的教学重点。**

### 9.1 手写版 vs 框架版必须尽量建立映射

本仓库几乎每个范式都有两条线：

- 手写版
- Spring AI Alibaba 框架版

所以注释不应只解释局部逻辑，还应尽量解释“它在对照线里的位置”。

例如：

- 手写版的 `while + switch` 对应框架版的 `StateGraph + addConditionalEdges`
- 手写版的 POJO 状态对应框架版的 `OverAllState`
- 手写版显式 coordinator 对应框架版的 `FlowAgent` / `SupervisorAgent`

### 9.2 要清楚区分业务状态和运行时控制状态

例如：

- `finalAnswer`、`currentCode` 这类是业务结果
- `stepStatus`、`search_failed`、`iteration_count` 这类更偏运行时控制

注释必须尽量帮助读者建立这个区分。

### 9.3 要把框架字段从“API 字段”解释成“运行时语义”

对下面这些字段，不要只写参数说明，要写它们在运行时里的作用：

- `outputKey`
- `instruction`
- `includeContents`
- `returnReasoningContents`
- `CompileConfig`
- `Saver`

### 9.4 要把“谁有权结束流程”讲清楚

这是多智能体模块里最容易让初学者混乱的点。

例如：

- Conversation 里谁有权决定结束
- CAMEL 里是谁发出终止标记
- Supervisor 里是主管结束，还是 Worker 结束
- Graph Flow 里是状态到 `END`，不是“模型觉得差不多了”

这些都应通过注释明确下来。

## 10. 测试代码注释规范

本仓库测试不是普通回归测试，很多测试同时承担**范式教学入口**的职责。

所以测试注释应优先解释：

- 这个测试在钉什么关键行为
- 为什么这个行为能代表该范式
- 手写版和框架版在这里对照了什么

不推荐只写：

```java
/**
 * 测试成功路径
 */
```

更推荐写成：

```java
/**
 * 验证手写版搜索成功时会进入 AnswerNode，而不是直接结束流程。
 *
 * 这个测试钉住的是 Graph Flow 最关键的行为之一：
 * 节点只负责更新状态，真正的下一跳由状态机统一决定。
 */
```

---

## 11. 反模式清单

以下写法在本仓库里应尽量避免：

- 用注释翻译变量名
- 对 getter/setter 逐个写注释
- 行尾塞入关键机制说明
- 一段简单代码写 5 行废话
- 只写“做了什么”，不写“为什么这样设计”
- 不解释上游 / 下游关系
- 不解释状态字段的读写者
- 在方法体内部大量滥用 Javadoc 注释

---

## 12. 收尾检查清单

在你认为“注释已经补好了”之前，至少再检查一遍：

1. 这段代码最难懂的运行时切换点，是否已经有解释
2. 状态字段的写入方和读取方，是否已经能从注释里看出来
3. 手写版和框架版的关键映射，是否已经尽量建立
4. 注释有没有解释设计意图，而不是只重复代码
5. 注释量是否失控，影响了代码可读性

最终目标不是“注释越多越好”，而是：

**让第一次接触 Agent Runtime 的 Java 工程师，能顺着注释把整条执行链真正看通。**
