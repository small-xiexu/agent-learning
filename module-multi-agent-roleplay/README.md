# module-multi-agent-roleplay

## 模块定位

`module-multi-agent-roleplay` 用于承载 CAMEL 风格的角色扮演与自主涌现范式。  
它关注的不是专家之间随意对话，而是通过严格角色设定与沟通协议，让协作能力“自然涌现”。

这个模块非常适合承载流水线式协作：

- 产品经理定义需求
- 架构师拆解方案
- 程序员实现
- 评审者检查结果

每个角色只做自己那部分高价值工作。

## 理论背景

CAMEL 风格范式的关键思想是：  
如果为多个智能体设置互补角色、明确目标与对话协议，高质量协作会自然出现。

这套范式依赖两个关键点：

### 1. 角色互补

不同 Agent 应拥有不同视角、不同职责、不同评价标准，而不是同一个助手的多个副本。

### 2. 引导性提示（Inception Prompting）

系统通过强约束 Prompt 提前规定：

- 你是谁
- 你该做什么
- 你不该做什么
- 你如何把结果交给下一角色

当角色协议足够清晰时，多 Agent 就会表现出类似团队协作的流水线特征。

## 运行机制

该模块建议采用“角色链 + 输出接力”的运行机制。

### 1. 角色定义

先为每个 Agent 定义明确角色，例如：

- 需求提出者
- 产品经理
- 解决方案设计者
- 程序员
- 评审者

### 2. 协议约束

每个角色的 Prompt 必须明确规定：

- 输入是什么
- 输出格式是什么
- 交付对象是谁

### 3. 顺序接力

前一个角色的输出不是给最终用户，而是给下一个角色继续消费。  
因此，每一步输出都应具备“可被下游消费”的结构性。

### 4. 停止与收敛

最后一个角色负责输出最终结果，或者把结果再交给评审角色做末端校验。

## Spring AI Alibaba 映射

这个模块与 Spring AI Alibaba 官方文档中的 `SequentialAgent` 模式高度对应。

官方文档明确说明：

- 多个 Agent 按预定义顺序依次执行
- 前一个 Agent 的输出成为下一个 Agent 的输入
- `outputKey` 用于把前序结果写入状态
- 后续 Agent 可以在 `instruction` 中通过 `{outputKey}` 占位符读取结果

这恰好就是角色扮演协作的最自然技术载体。

### 关键映射点

- `SequentialAgent`：承载角色流水线
- `instruction` / `systemPrompt`：承载 Inception Prompting
- `outputKey`：承载角色间交接协议
- `includeContents` / `returnReasoningContents`：控制角色是否看到前一角色的完整推理过程

## 与 framework-core 的关系

Roleplay 模块依赖 `framework-core` 提供两个关键统一层：

- `Message / Memory`：保存角色间交互历史
- `HelloAgentsLLM`：统一模型调用，避免角色实现绑死底层模型 API

建议把“角色契约”抽象为显式对象，例如：

- `RoleProfile`
- `RoleContract`
- `StageOutput`

这样角色协作就不只是 Prompt 文案，而是模块内可管理的领域对象。

## 工程落地建议

### 1. 角色必须有冲突最小化设计

角色互补不等于角色冲突。  
如果两个角色都试图做“总体决策”，流程会开始重复和拉扯。

### 2. 输出必须为下游优化

不要让上游角色输出一大段难以消费的自然语言。  
每一阶段输出都应该为下一阶段服务。

### 3. Prompt 要定义边界，而不是只定义人设

“你是资深产品经理”远远不够。  
必须进一步约束输入、输出、格式、禁止项和交接规则。

### 4. 控制角色漂移

长链路角色协作很容易出现“后面的角色开始重新做前面的事”。  
应通过系统提示和阶段契约严格约束。

## 适用场景与边界

### 适用场景

- 流水线式协作任务
- 需求 -> 方案 -> 实施 -> 评审 这类角色链任务
- 希望通过明确角色协议提升输出质量的场景

### 不适合的场景

- 需要自由切换控制权的群聊式协作
- 分支和循环很多的复杂状态机
- 角色边界本身不清晰的任务

## 结论

角色扮演范式的本质，不是“给 Agent 起不同名字”，而是通过严格角色契约让协作涌现。

在本项目中，`module-multi-agent-roleplay` 应承担角色链协作的标准实现，为顺序执行、Prompt 协议和输出接力提供清晰的工程落点。
