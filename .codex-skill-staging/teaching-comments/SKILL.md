# Teaching Comments

## 概览

这个 Skill 用来给代码补**教学型注释**，目标不是把代码变得更啰嗦，而是让学习者真正看懂：

- 这段代码为什么存在
- 它在运行时到底怎么流转
- 它和周边类、字段、状态之间是什么关系

核心原则：

**解释“为什么这样设计、运行时如何流动”，而不是把显而易见的语法再翻译一遍。**

## 什么时候使用

当用户出现下面这些意图时，使用这个 Skill：

- 明确要求“补注释”“加详细解释”“把代码讲清楚”
- 说代码虽然能跑，但看起来一知半解
- 正在阅读框架代码、工作流代码、状态机代码、多步骤编排代码
- 卡在构造器重载、状态流转、路由判断、生命周期、控制流这些点上

不要在这些场景使用：

- 自解释的 CRUD 代码
- getter/setter 一类样板代码
- 仅仅因为命名差，就想用注释替代重命名
- 机械地给每一行都补注释

## 注释目标

好的教学型注释，应该帮助读者回答这 5 个问题：

1. 这个类或方法在整个系统里扮演什么角色？
2. 它的输入从哪里来？
3. 它的输出会被谁消费？
4. 这段代码执行后，运行时接下来会发生什么？
5. 为什么这里要这样设计，而不是写成更表面的形式？

## 工作流

补注释前：

1. 先找出真正让人困惑的点，是运行时行为、抽象层级，还是隐藏的数据流。
2. 先画清这段代码的上游和下游关系。
3. 优先注释边界、切换点、设计意图，不要优先注释局部语法。

补注释时：

1. 为关键类补类级 Javadoc，说明角色和协作关系。
2. 为关键方法补方法级 Javadoc，说明输入来源、输出含义和运行时效果。
3. 为共享状态、路由键、控制字段补字段注释。
4. 在非直觉代码块前加简短块注释。

补注释后：

1. 删除只是“把代码翻译成中文”的低价值注释。
2. 检查一个初学者是否能不跳文件就理解主流程。
3. 保持简洁，不让注释量压过代码本身。

## 必补的注释类型

### 1. 类级角色注释

对核心类，至少解释：

- 它是什么
- 它解决什么问题
- 它是入口、协调器、节点、路由器、状态容器，还是适配器
- 它和相邻关键类是什么关系

优先使用这类表达：

- `职责：...`
- `运行时角色：...`
- `上游输入：...`
- `下游输出：...`
- `对照关系：...`

说明：

- 类、方法、字段这类“声明级别”的说明，优先使用 `/** ... */`
- 方法体内部的代码块说明，优先使用 `//` 或 `/* ... */`
- 不要在方法体内部大量使用 Javadoc 风格注释，避免让读者误以为那是声明说明而不是执行逻辑说明

### 2. 方法级运行时注释

对关键方法，重点解释：

- 谁会调用它
- 它是入口还是中间步骤
- 它会产生什么副作用
- 返回值或写入状态最终会被谁消费

### 3. 字段级状态注释

对共享状态、控制标记、路由字段，重点解释：

- 谁写入它
- 谁读取它
- 它是业务数据还是运行时控制数据
- 为什么必须存在这个字段

### 4. 非直觉代码块注释

这些地方优先加块注释：

- 构造器链式调用
- 状态切换
- 条件路由
- handoff / 交接逻辑
- 重试、回环、终止条件
- 手写版和框架版的一一映射

## 高优先级补注释目标

优先处理这些地方：

- 状态对象和状态键
- coordinator / router / graph / workflow 一类核心类
- `outputKey`、`instruction`、`includeContents`、`CompileConfig` 这类框架字段
- 构造器重载和默认依赖注入
- 终止条件、最大轮次、递归/回环限制
- 那些用来钉住“范式关键行为”的测试方法

## 注释形式选择

按下面这个规则选注释形式：

- 说明“这个东西是什么”：用 `/** ... */`
- 解释“这段代码为什么这样跑”：用 `//` 或 `/* ... */`

推荐示例：

声明级别：

```java
/**
 * 图执行引擎
 *
 * 职责：用 while + switch 驱动手写版状态机流转。
 */
public class GraphRunner {
}
```

方法体内部：

```java
// 搜索失败后不直接抛异常，而是把失败事实写回状态，
// 让后续路由器决定是否进入 fallback 分支。
if (shouldFail) {
}
```

不推荐示例：

```java
/**
 * 搜索失败后的处理
 */
if (shouldFail) {
}
```

## 反模式

避免写这种注释：

- “给 x 赋值”
- “i 加 1”
- “调用 service 方法”
- 和方法名重复的注释
- 每行一条的流水账注释

应该把低价值注释替换成下面这些解释：

- 设计意图
- 运行时后果
- 为什么会有这个分支
- 初学者最容易误解什么

## 示例改写

弱注释：

```java
// 创建搜索工具
this.searchTool = new MockSearchTool(false);
```

更好的教学型注释：

```java
// 提供默认的成功路径依赖。
// 如果测试或演示需要走降级分支，应改用重载构造器，
// 显式注入 shouldFail=true 的搜索工具。
this.searchTool = new MockSearchTool(false);
```

## 收尾检查

结束前确认：

- 注释解释的是“机制”，不是只有“结果”
- 最容易让人困惑的运行时切换点已经可见
- 注释没有和命名打架
- 没有出现大段低信号废话
