# module-graph-flow-paradigm

## 模块定位

`module-graph-flow-paradigm` 用于承载以状态机和有向图为核心的复杂编排范式。  
它解决的问题不是“如何再包装一个 Agent”，而是“如何把复杂流程从隐式代码分支提升为显式图结构”。

当任务存在：

- 多分支条件路由
- 循环重试
- 人工介入
- 子图嵌套
- 长生命周期状态

时，Graph Flow 往往比单纯堆叠 Agent 更合适。

## 🎯 最佳实践场景

**带兜底机制的智能问答流水线**：利用图的“条件边”实现容错，例如搜索节点失败时，状态机不会崩溃，而是按条件流转至内部知识兜底节点。

## 理论背景

图编排范式的核心思想是：  
把系统执行过程建模为一个显式状态机。

在这个模型里：

- `Nodes` 代表计算步骤或处理单元
- `Edges` 代表状态流转关系
- `Conditional Edges` 代表基于状态判断的动态跳转
- `State` 代表流程运行中的事实集合

它的优点在于：

- 流程结构可视化
- 跳转规则可审计
- 状态变化可观察
- 可以自然表示循环、分支、回退和子流程

这比把所有流程塞进 `if/else + while` 更适合企业级复杂任务。

## 运行机制

该模块建议坚持“显式状态、显式节点、显式边”的设计原则。

### 1. 状态初始化

先定义流程状态模型，明确：

- 输入数据
- 中间变量
- 分支判断字段
- 完成标记

### 2. 节点执行

每个节点只做一件事：

- 调模型
- 调工具
- 聚合数据
- 判断条件
- 写状态

### 3. 边路由

节点执行结束后，不由代码隐式决定下一步，而由边规则决定：

- 固定边：总是跳到下一个节点
- 条件边：根据状态决定路由
- 循环边：回到某个节点继续纠偏

### 4. 结束条件

图应有明确终点，而不是“流程自己停下来”。  
终止节点和异常节点都应是显式的一等概念。

## Spring AI Alibaba 映射

这个模块与 Spring AI Alibaba 的映射是本项目里最直接、最底层的一类。

### 1. `FlowAgent`

官方文档明确给出了 `FlowAgent` 抽象基类。  
它是 `SequentialAgent`、`ParallelAgent`、`LlmRoutingAgent` 等流程型 Agent 的共同基类。

### 2. `StateGraph`

`FlowAgent` 的核心扩展点是：

- `buildSpecificGraph(FlowGraphBuilder.FlowGraphConfig config)`

这意味着流程不应写死在 Builder 参数里，而应通过 `StateGraph` 显式构建。

### 3. Graph Builder

Spring AI Alibaba 还提供了 `FlowGraphBuilder`、`CompileConfig` 等配套能力，帮助开发者把自定义图结构编译为可运行的流程。

因此，`module-graph-flow-paradigm` 的核心任务并不是“复用现成顺序流”，而是：

- 定义节点
- 定义边
- 定义状态
- 定义图编译与执行语义

## 与 framework-core 的关系

Graph Flow 模块仍然应站在 `framework-core` 的协议之上，而不是绕开它直接操作底层模型。

- `AgentLlmGateway`：让图中的模型节点统一调用入口
- `ToolRegistry`：让图中的工具节点统一执行入口
- `Memory / Message`：让对话上下文和过程消息有统一语义

这样做的价值在于：

- 图运行时负责“调度”
- `framework-core` 负责“协议统一”

两者边界清晰，后续更容易替换节点内部实现。

## 工程落地建议

### 1. 先定义状态，再定义节点

图编排里最容易失败的地方，不是边不够多，而是状态定义混乱。  
状态字段必须先定清楚，再去写节点逻辑。

### 2. 节点必须单一职责

一个节点如果既做检索、又做推理、又做路由判断，后面几乎无法调试。

### 3. 不要把“条件”写回提示词

分支判断应尽量体现在状态和条件边中，而不是埋在系统提示里让模型自己猜下一步。

### 4. 图结构要为可观测性服务

流程图不是为了“更酷”，而是为了让流程路径可追踪、可调试、可回放。

## 适用场景与边界

### 适用场景

- 长流程工作流
- 多分支、多循环、多状态任务
- 对可观测性和可治理性要求高的企业编排

### 不适合的场景

- 只需一轮或两轮工具调用的简单 Agent
- 状态简单、延迟敏感的轻量任务

## 结论

Graph Flow 的价值不在于“它比 ReAct 高级”，而在于它让复杂流程从隐式逻辑变成显式结构。

在本项目中，`module-graph-flow-paradigm` 应承担复杂编排底座的角色，是从 Agent 走向 Workflow 和状态机系统的关键桥梁。
